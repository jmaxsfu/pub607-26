# Textual Collaboration: Git and Github

How will all of us collaborative on our project? One possibility would be for us to all decide on a naming convention for our files, and then email all of our work around to each other endlessly, and hope for the best. What could go wrong?

There are better ways of working. Software developers for decades have worked in large, sometimes complex teams, and -- as you now know -- they're working mostly on text files. As a result, software developers have evolved some *really* sophisticated tools for managing collaborative work on a set of files.

The state of the art currently (since around 2007) is a tool called **Git** and a sharing platform called **Github**. Please note that these two things are *not the same thing*, even though people mostly just say Github as if it were synonymous with Git.

**Git** is software. It is a "distributed version control system" -- which means it is a file repository that allows multiple people to have access to parts of it ("distributed"), while it keeps track of all the changes ("version control") that people make. The result is that multiple people can be working on the same set of files at the same time without really having to worry about each other, and the system makes sure that chaos doesn't ensue with people overwriting each other's work. It does so by fine-grained tracking of changes (or "diffs" in Unix parlance) to text files.

You can use Git software all by itself (indeed I have done so, and continue to do so with small projects like my own blog), but for larger multi-person projects, it helps to have a host system where everything lives, where people can make user accounts, and project managers can organize who has access to what.

**Github** is, for the most part, the system that everyone uses. According to Wikipedia (Feb 2025), Github hosts "100 million developers and more than 420 million repositories, including at least 28 million public repositories." That's huge. Would you be surprised to learn that this is all owned by Microsoft? They bought Github in 2018. Would you be surprised to hear that the entire planet-sized repository is used to train AI? 

While it was designed and built for software developers, it works perfectly well for any project involving text files: website development and ebook development certainly! 

What if we don't want to use GitHub? Are there alternatives? Of course there are!

## How Git works

Git is a complex piece of software, and it does many very sophisticated things. We are going to only use a tiny fraction of it: *the parts that I actually understand*, which is minimal!

The basic idea is that a **repository** (a folder full of files) can be "cloned" by a participant (like you); when you **clone** a repository (which is generally the first thing you do when you join a project), you are essentially downloading a copy of it to your own computer -- but the copy you download *maintains a connection* to the remote version, and Git keeps track of any changes that exist between the two copies, whether those changes happened on your end, or on the original repository. All of us can keep our own repositories (**repos**) and Git will keep us all in sync.

We manage changes by two basic operations: **push** and **pull**. Pushing means merging *your changes* with the remote repository; pulling means merging *remote changes* with your local version.

One more important detail: you must explicitly **commit** any change you make, normally by writing a tiny summary saying what the change was: "Fixed a typo in the title!" -- or whatever it might be.

Git was originally designed as a Unix tool, run by typing commands in the shell. But Github makes available a nice easy tool called **Github Desktop** that has nice friendly buttons for "Pull," "Commit," "Push," that make the whole thing a lot friendlier.

---

The usual sequence, when you sit down to work, goes like this:

1) **Pull** the latest version of the repository into your local version, so that you're up to date. Github uses the term **Fetch** here too, which just means checking for any remote changes (if there are any changes, you'll be prompted to **Pull** them).

2) You do your work locally, editing files, saving files, etc. When you save a file, Git notices; in Github Desktop, you will see that a file has been changed (and even what the change was).

3) When you've completed a meaningful change (i.e., when it's worth making a comment describing what you did), you add a note about your change and push the **Commit** button. A single "commit" might involve many changes to many files, but conceptually, it is *one change you're going to take responsibility for*.

4) Then you **Push** your committed change from your local files to the main repository. Now you're in sync again, and everyone else can build on your work.

5) It is still possible to get tangled up, if two people edit the same line at the same time. For this reason, you should get into the habit of click the **Fetch origin** button before you **Commit**, just to check. Fear of commitment has its place.

There's oodles and oodles more things Git can do (much of which I don't use or even understand), but what I've laid out here is mostly what we will do. This allows us all to work on our own computers on our own time, and still keep in sync with everyone else.
