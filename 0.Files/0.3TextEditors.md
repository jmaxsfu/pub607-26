## Text Editors

Consider the text editor. It edits text. That's all it does. 

In a system made up of text files, a text editor is the core tool.

There are many text editor apps out there -- they all work more or less the same. For PUB 607 I am recommending [Sublime Text](https://www.sublimetext.com/download), which works identically on both Mac and Windows, and which is, well, *sublime*.

It looks a little bit like a word processor, in that there's a window you can type words in, but it is actually very different. 

A word processor isn't really about processing words; it's about producing printed pages -- generally in an office communications paradigm. A word processor has zillions of features for formatting and automating and making things complicated.

![](../img/word-hell.png "Word hell")

A text editor just edits text. It opens up a text file, shows you what's inside, and lets you edit it.

---

*Text is Special.* This goes deeper than just the mechanics of files and file formats. 

Text is special because it's uniquely editable, changeable, mutable. 

But, we also like to think that text is special because it's unchanging! Not quite, though: that's *print*, not *text*. 

The "text" as an idea comes into being at the point when Gutenberg makes the type "moveable" and editors have to think about what the text *really* is, before they consign it to the printing press.

*Editors* are the people who think about the text. It should be no surprise that I think if you're an editor of text you should use a text editor to edit your text. 

---

Your text editor has lots of menus and features; these are almost all either about moving around in the text or about interacting with outside systems: with the filesystem and file management and programming language environments; these tools are built largely for programmers. But the tool itself is very simple: it shows you a text file and allows you to edit it.

The brilliant thing about a text file is that *what you see is all there is*. There are no hidden layers, nothing hiding behind the scenes. It's just letters and numbers and punctuation, arranged in **lines**. 

*Lines* are important; the original design of Unix put a lot of emphasis on lines. Your text editor helps you see where the lines are by numbering them, in the left margin.

Note that there's a difference between a visual line and a 'logical' line -- the latter being ended with a Return/Newline character -- the thing that happens when you hit the *Return* key on your keyboard. A logical line might be a whole paragraph long; the text editor can show it to you 'wrapped' like in a word processor, or unwrapped, in which case you'd need scroll sideways to get to the end of it.

Your text editor also sometimes shows you colours.

This is a friendly thing it does: by recognizing what *kind of text*  you're looking at -- in other words, what the meaningful structures are in it -- it will show you in colour where some of those structures begin and end. That's helpful.

The important thing is that the colour *isn't in the file*. The text in the file has no colour; it's just text. But the editor helps you out by adding colour where it sees it.

How does it do that? By **parsing** the text.

Parsing, which is a term from linguistics, means taking a sentence apart into its grammatical parts; in computing it means something quite similar: usually not with sentences, but other kinds of text strings. A *huge* portion of the nuts and bolts of computing is about parsing; how else would a computer figure out what you were trying to get it to do?

A programming language (like Javascript) has to be parsed into machine-readable structures. A markup language (like HTML) has to be parsed into its component structures. Your editor parses a text file in order to colour the structures of the text to help you visually navigate it.
